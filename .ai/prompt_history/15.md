## Plan: `create-endpoint.prompt.md` Prompt Dosyası Oluşturma

Mevcut servislere (product-service / order-service) tek veya CRUD seti halinde yeni endpoint eklemek için standart bir prompt şablonu oluşturulacak. Prompt, AGENTS.MD'nin "Contract First" ve "Uydurma YOK" kurallarına uygun şekilde girdileri toplar, OpenAPI spec'i önceliklendirir, ardından implementasyon adımlarını sıralı batch'ler halinde yürütür. Yeni claim gerekiyorsa add-claim.prompt.md'ye yönlendirir; entity var kabul edilir.

**Steps**

1. .github/prompts/create-endpoint.prompt.md dosyası oluşturulacak. YAML frontmatter'da `agent: agent` ve Türkçe `description` bulunacak — mevcut add-claim.prompt.md ve protect-endpoint.prompt.md formatına birebir uyumlu olacak.

2. **Girdi Parametreleri** bölümü aşağıdaki alanları toplayacak:

   | Alan | Açıklama | Zorunlu |
   |---|---|---|
   | **Servis** | product-service \| order-service | Evet |
   | **Kaynak (Resource)** | Örn. `products`, `orders` | Evet |
   | **Mod** | `tek endpoint` \| `CRUD seti` | Evet |
   | **Endpoint(ler)** | Her biri için: HTTP metot + path + operationId + kısa açıklama | Evet |
   | **Public mi?** | Her endpoint için: Evet/Hayır (D010 referansı) | Evet |
   | **Claim** | Her korumalı endpoint için: mevcut D012 claim adı veya "YENİ" | Evet (korumalıysa) |
   | **Request Body Şeması** | Alan adı, tip, zorunluluk, constraint (POST/PUT/PATCH için) | Koşullu |
   | **Response Şeması** | Mevcut bir şema referansı veya yeni şema tanımı | Evet |
   | **Pagination** | Sayfalı sonuç dönecek mi (GET listeler için) | Koşullu |
   | **Hata Kodları** | Domain-spesifik hata kodları ve HTTP status eşlemeleri | Evet |
   | **Feign Çağrısı** | Başka servise çağrı gerekiyorsa hedef servis + endpoint | Hayır |

3. **Adım 1 — Ön Kontrol**: DECISIONS.MD okunacak. D009'da endpoint+metot çakışma kontrolü, D010'da public endpoint tutarlılığı, D012'de claim varlığı doğrulanacak. "YENİ" claim varsa add-claim.prompt.md'ye yönlendirme yapılıp DUR denecek.

4. **Adım 2 — OpenAPI Spec Güncelleme** (Contract First): İlgili spec dosyasına (product-service-v1.yaml veya order-service-v1.yaml) yeni path + şema tanımları eklenecek. CRUD setinde tüm endpoint'ler tek seferde spec'e yansıtılacak. Security alanı kuralları (D010 public → `security: []`, diğerleri → `bearerAuth`) uygulanacak. Mevcut spec'teki kalıplara (pagination şeması `*Page`, `ErrorResponse`, UUID id, validation constraint'leri) uyum zorunlu.

5. **Adım 3 — DTO Oluşturma**: Girdi şemasına göre `dto/request/*Request.java` ve `dto/response/*Response.java` dosyaları oluşturulacak. Mevcut kalıp: Jakarta validation anotasyonları request'te, plain POJO response'ta, Lombok yok.

6. **Adım 4 — Service Katmanı**: `service/*Service.java` interface'ine ve `service/impl/*ServiceImpl.java` implementasyonuna yeni metot(lar) eklenecek. `@Transactional(readOnly = true)` read ops için, `@Transactional` write ops için.

7. **Adım 5 — Repository Katmanı**: Yeni sorgu gerekiyorsa `repository/*Repository.java`'ya Spring Data method veya `@Query` eklenecek.

8. **Adım 6 — Mapper Güncelleme**: `mapper/*Mapper.java`'ya yeni `toResponse`, `toEntity`, `toPage` vb. metotlar eklenecek (manuel mapping, MapStruct yok).

9. **Adım 7 — Controller Metot Ekleme**: `controller/*Controller.java`'ya yeni metot eklenecek. `@PreAuthorize` korumalı endpointlere uygulanacak (D009 referansı). Public endpointlere `@PreAuthorize` eklenmeyecek. POST için `Location` header + `201 Created`, DELETE için `204 No Content`, GET/PUT/PATCH için `200 OK` kalıbı.

10. **Adım 8 — Exception Handling**: Yeni domain exception gerekiyorsa `exception/` paketine eklenecek, `GlobalExceptionHandler`'a handler eklenecek.

11. **Adım 9 — SecurityConfig Kontrolü**: Public endpoint eklendiyse `SecurityConfig.java` (ilgili servis + gateway) güncellenerek `permitAll()` listesine eklenecek. D010 güncellemesi için kullanıcıya DECISIONS.MD'ye ekleme onayı sorulacak.

12. **Adım 10 — Doğrulama Tablosu ve Happy-path**: Her endpoint için beklenen HTTP status senaryoları tablosu (token var/yok, doğru/yanlış claim) ve çalıştırılabilir `curl` komutu verilecek.

13. Tüm adımlar AGENTS.MD Bölüm 1.2'deki **5'er dosyalık batch** kuralına tabi olacak — prompt içinde batch sınırları açıkça belirtilecek ve her batch arası onay istenecek.

14. **Çıktı Formatı** bölümü mevcut prompt'larla aynı yapıda olacak: Dosya Dökümü (yol yok), Özet, Sorular (varsa), Happy-path test.

**Verification**
- Prompt dosyasının YAML frontmatter'ı geçerli olmalı (`agent: agent`, `description: ...`).
- Tüm governance referansları (D009, D010, D012) doğru atıflanmalı.
- Mevcut prompt'larla (`add-claim`, `protect-endpoint`) yapısal tutarlılık kontrol edilmeli.
- Claim gerektiren bir senaryo için prompt adımları izlendiğinde add-claim.prompt.md'ye yönlendirme olduğu doğrulanmalı.
- CRUD seti modunda girdi parametreleri bölümünün birden fazla endpoint kabul ettiği doğrulanmalı.

**Decisions**
- CRUD seti desteği: Tek promptta birden fazla endpoint tanımına izin verilecek, her endpoint ayrı satırda girilecek.
- Claim delegasyonu: Yeni claim gerektiğinde prompt DUR deyip add-claim.prompt.md'ye yönlendirecek.
- Entity scope dışı: Prompt yeni entity/kolon oluşturmayı kapsamaz; entity zaten var kabul edilir.
- Dosya adı: `create-endpoint.prompt.md`.
