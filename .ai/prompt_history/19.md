## Mevcut Durum Analizi

### Git Geçmişinde Tespit Edilen Sorunlar

| Sorun | Örnek |
|---|---|
| Tek branch üzerinde çalışma | Yalnızca `master` branch'i var, feature branch yok |
| Tutarsız commit mesaj formatı | `feat(*): add product...`, `add auth-service`, `fix login problem`, `.` |
| Anlamsız commit | `09d7624` → `.` |
| Tekrarlı/belirsiz mesajlar | 3x `update agents.md`, 2x `update governance` |
| Dil tutarsızlığı | İngilizce + Türkçe karışık (`create-endpoint-prompt eklendi`) |
| Scope/context eksikliği | `fix login problem` — hangi servis? ne düzeltildi? |
| PR/code review süreci yok | Tüm commitler doğrudan `master`'a push edilmiş |

### Önerilen Best Practice'ler

#### 1. Branching Stratejisi (Git Flow - Simplified)

```
master          ← production-ready, doğrudan commit YASAK
  └── develop   ← entegrasyon branch'i
       ├── feat/product-service-crud
       ├── feat/order-service-sync
       ├── fix/auth-login-redirect
       └── chore/update-governance
```

| Branch Prefix | Kullanım |
|---|---|
| `feat/` | Yeni özellik |
| `fix/` | Bug düzeltme |
| `chore/` | Yapılandırma, dokümantasyon, CI/CD |
| `refactor/` | Kod yeniden yapılandırma |

#### 2. Conventional Commits Formatı

```
<type>(<scope>): <kısa açıklama>

[isteğe bağlı gövde]

[isteğe bağlı footer]
```

**Mevcut commit'lerin olması gereken halleri:**

| Mevcut | Olması Gereken |
|---|---|
| `.` | *(bu commit yapılmamalıydı)* |
| `add auth-service` | `feat(auth): initialize auth-service module` |
| `fix login problem` | `fix(auth): resolve token validation on login endpoint` |
| `update agents.md` | `chore(docs): update AGENTS.MD governance rules` |
| `add gateway` | `feat(gateway): add Spring Cloud Gateway module` |
| `create-endpoint-prompt eklendi` | `chore(docs): add create-endpoint prompt template` |
| `update to hexagonal arch.` | `refactor(product): migrate to hexagonal architecture` |

**Type kümeleri:** `feat`, `fix`, `refactor`, `chore`, docs, `test`, `ci`, `perf`

#### 3. PR (Pull Request) Kuralları

- Her feature/fix branch'i `develop`'a PR ile merge edilmeli.
- PR açıklamasında: ne yapıldı, neden yapıldı, nasıl test edilir bilgisi bulunmalı.
- `master`'a merge yalnızca `develop`'dan ve release hazır olduğunda yapılmalı.
- Squash merge tercih edilmeli (temiz geçmiş için).

#### 4. Dil Tutarlılığı

Tüm commit mesajları, branch adları ve PR başlıkları **tek bir dilde** (tercihen İngilizce) yazılmalı. Türkçe-İngilizce karışımı (`create-endpoint-prompt eklendi`) kullanılmamalı.
